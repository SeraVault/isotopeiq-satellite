IsotopeIQ Satellite

Satellite is an open source platform for configuration baseline collection, drift detection, and scripted remediation across heterogeneous device environments. It connects to managed devices over SSH, WinRM, Telnet, or a lightweight agent, runs user-defined scripts to collect configuration state, normalises that output into a canonical JSON structure, and compares subsequent runs against the stored baseline to surface any changes.

Deployment scripts can be pushed to devices on a schedule or on demand, providing Ansible-like remediation capability from the same interface.

How It Works

Satellite organises its work around policies. A policy binds one or more devices to a script job and a collection schedule. When the policy runs, Satellite connects to each device, executes the collection script, passes the output through a parser script to produce a canonical JSON snapshot, compares that snapshot to the stored baseline, and fires any configured notifications if differences are found.

1

Connect

Satellite connects to the device using the protocol configured for that device: SSH, WinRM, Telnet, HTTPS/API, or by polling a lightweight agent running on the device on port 9322.

→
2

Collect

A collection script runs on the remote device (or on the Satellite server for API-based devices) and returns raw text output — command output, config file contents, registry values, etc.

→
3

Parse

A parser script runs on the server, receives the raw output on stdin, and writes a populated canonical JSON object to stdout. The output is validated against the canonical schema before being stored.

→
4

Baseline & Diff

The canonical snapshot is stored as the device's baseline on first run. On subsequent runs, a deep structural diff is computed against the stored baseline. Any differences become drift events.

→
5

Notify & Remediate

Drift events trigger configured notifications (syslog, email, FTP/SFTP). Separately, deployment scripts can be pushed to any device on demand or on schedule to correct configuration or enforce standards.

Canonical Schema

All parser output — regardless of device type or collection method — is normalised into the same 12-section JSON structure. This means drift comparisons, audit queries, and SIEM exports have a consistent shape whether you're looking at a Linux server, a Windows host, a Cisco router, or a legacy system.

The schema is enforced by JSON Schema validation on every collection run. If a parser produces output that doesn't conform, the run fails cleanly and the existing baseline is preserved.

The custom section is unconstrained and is available for device-specific or application-level data that doesn't fit the standard sections.

A volatile rules engine allows specific fields to be excluded from drift comparison — useful for counters, uptime values, or any field that changes on every run by design.

canonical JSON structure
{
  "device":             { /* hostname, type, location */ },
  "hardware":           { /* CPU, RAM, serial */ },
  "os":                { /* name, version, kernel */ },
  "network":            { /* interfaces, routes, DNS */ },
  "users":              [ /* local accounts */ ],
  "groups":             [ /* local groups and members */ ],
  "installed_software": [ /* installed packages/apps */ ],
  "services":           [ /* running/configured services */ ],
  "packages":           [ /* package manager state */ ],
  "filesystem":         [ /* mounts, permissions */ ],
  "security":           { /* firewall, audit, policy */ },
  "custom":             { /* device-specific data */ }
}

Capabilities

Device & Credential Management

  • Register any reachable host — Linux, Windows, macOS, network appliances, SNMP systems, legacy platforms
  • Connection types: SSH, WinRM/NTLM, Telnet, HTTPS/REST API, Agent Pull (port 9322)
  • SSH host key pinning per device (MITM protection)
  • Credentials stored encrypted at rest (Fernet). Passwords and private keys are write-only — they cannot be retrieved from the UI after saving
  • Credential types: SSH password, SSH key (Ed25519 / ECDSA / RSA), WinRM, API token
  • Credentials are shared across devices and injected at runtime via {{PLACEHOLDER}} substitution — never embedded in script text

Scripts & Script Jobs

  • Three script types: Collection (runs on the remote device), Parser (runs on the server, stdin → canonical JSON), Deployment (pushed to the device for remediation or hardening)
  • Script languages: Shell, Python, PowerShell, VBScript, SQL
  • Scripts can run on the client (uploaded and executed on the device) or on the server (useful for API-based collection — Cisco IOS, SNMP, etc.)
  • A Script Job chains multiple scripts into an ordered pipeline, with output piped between steps
  • In-browser script editor with syntax highlighting (CodeMirror)

Policies & Scheduling

  • A policy binds a set of devices to a script job and a collection schedule
  • Schedule options: hourly, daily, weekly, monthly, or raw cron expression
  • Collection methods: script execution (SSH/WinRM/Telnet) or agent pull (HTTP GET to agent on port 9322)
  • Configurable delay between devices to throttle job pacing
  • "Run Now" and "Deploy Now" manual triggers available per policy

Baselines

  • One canonical JSON snapshot stored per device, updated on every successful collection run
  • Viewable in the UI with expandable sections per schema category
  • On first successful run, the snapshot is stored as the initial baseline with no drift event raised
  • Acknowledging a drift event promotes the current state as the new baseline and records the reason and username

Drift Detection

  • Deep structural diff computed by deepdiff between the current canonical snapshot and the stored baseline
  • Drift event lifecycle: new → acknowledged → resolved
  • If a subsequent run shows the device has returned to baseline, the event is auto-resolved
  • Volatile rules suppress specified fields from comparison — five rule types: section_field, item_field, nested_field, exclude_key, exclude_section
  • Rules take effect within 60 seconds, no restart required

Remediation

  • Deployment scripts can be pushed to any device or group of devices, either on a schedule via a policy or triggered manually
  • Scripts run in the same execution context as collection scripts (SSH, WinRM, etc.) using the same credential system
  • Use cases: enforce hardening standards, revert unauthorised changes, install or remove packages, update configuration files

Job Monitor

  • Real-time view of running, pending, and historical jobs with per-device status
  • Per-device job results show raw collected output, parsed canonical JSON, any error messages, and an inline drift diff
  • Jobs can be cancelled mid-flight

Notifications

  • Per-policy Post-Collection Actions define trigger (new baseline / drift detected / always) × destination (syslog / email / FTP/SFTP)
  • Syslog messages include device name, policy name, and a summary of changed fields — suitable for SIEM ingestion
  • A global fallback action handles agent-push jobs not associated with a policy

Authentication & Access

  • Local username/password with JWT access and refresh tokens
  • LDAP / Active Directory: group-to-role mapping, configurable via UI or environment variables
  • SAML 2.0 SP-initiated SSO: standard metadata endpoint at /saml2/metadata/, compatible with any standards-compliant IdP

Audit Log

  • All significant actions are recorded automatically via middleware — no per-view instrumentation needed
  • Each record includes: timestamp, username, action type, resource type/ID/name, HTTP method, request path, response status, source IP
  • Actions covered: create, update, delete, login, logout, manual triggers

Agent

  • Pre-compiled single-binary agents for Linux x86-64, Linux x86, macOS, and Windows
  • Agent exposes GET /collect returning canonical JSON. Satellite polls it on schedule
  • Authenticated via X-Agent-Secret shared header
  • Python source equivalents available for environments blocking compiled binaries or for air-gapped installation
  • Installer scripts provided for Linux (systemd), macOS (launchd), and Windows (scheduled task + firewall rule)

Data Retention

  • Configurable retention windows per category: raw collected data, parsed canonical data, job history, logs
  • Defaults: raw 90 days / parsed 365 days / job history 180 days / logs 90 days
  • Daily pruning task runs at 03:00 UTC. Set any value to 0 for indefinite retention

Deployment

Satellite runs as a Docker Compose stack (6 containers: PostgreSQL, Redis, Celery worker, Celery Beat scheduler, Django/Gunicorn backend, Vue/Vite frontend). A bare-metal deployment path is also included for Ubuntu 24.04 LTS hosts using systemd and nginx.

Docker Compose

Requires Docker and Docker Compose on any Linux host.

terminal
$ git clone https://github.com/SeraVault/isotopeiq-satellite
$ cd isotopeiq-satellite
$ cp .env.example .env
# edit .env — set SECRET_KEY, FERNET_KEY, etc.
$ ./deploy.sh up

Bare Metal

The included deploy/push.sh script rsyncs the project to a remote Ubuntu 24.04 host and configures systemd services and nginx. TLS options: self-signed, Let's Encrypt, or bring your own certificate.

Stack Components

  • Backend: Python / Django 4.2 + Django REST Framework
  • Task queue: Celery 5 with Redis broker and result backend
  • Scheduler: Celery Beat with django-celery-beat (DB-backed schedules)
  • Database: PostgreSQL 16
  • Frontend: Vue 3, Vuetify 3, Pinia, Vue Router, Axios, Vite

Air-Gapped Environments

All agent binaries are pre-compiled and included in the repository. Python source agents are provided as a fallback. No external network access is required at runtime — there are no cloud dependencies or telemetry calls.
